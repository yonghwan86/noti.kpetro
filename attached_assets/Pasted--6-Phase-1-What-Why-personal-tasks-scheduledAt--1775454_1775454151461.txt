

# 이슈 #6 — 내 일정 기간 등록 (Phase 1)

---

## What & Why

현재 개인일정(personal_tasks)은 `scheduledAt` 하나의 시점만 등록할 수 있어, "4/7~4/11 상반기 장비 점검" 같은 며칠에 걸친 업무를 표현할 수 없습니다. 사용자가 기간 업무를 등록하려면 매일 별도 일정을 만들어야 하는 불편이 있습니다.

`scheduledEndAt` 컬럼을 추가하여 시작일~종료일 기간 등록을 지원합니다. 기존 단일 날짜 일정은 `scheduledEndAt = null`로 동작하므로 하위 호환이 완벽합니다.

### 이슈 #5 (다이제스트 통합)와의 연관

이슈 #5에서 morningNotified 플래그는 boolean 방식이 유지되었습니다. 기간 일정이 도입되면 첫날에 `morningNotified = true`가 설정된 후 이후 날에는 알림이 발송되지 않는 문제가 발생합니다. 이를 해결하기 위해 `morningNotified`를 boolean에서 `lastMorningNotifiedDate`(text)로 변경합니다.

---

## Done Looks Like

- 일정 생성/수정 시 종료일(scheduledEndAt) 선택 가능 — 선택하지 않으면 기존과 동일한 단일 날짜 일정
- 기간 일정이 캘린더/목록에서 해당 기간 전체에 걸쳐 표시됨
- 6AM 다이제스트에서 기간 일정이 기간 중 매일 "오늘 일정"에 포함됨
- 모닝 푸시도 기간 중 매일 발송됨
- 10분 전 리마인더는 시작일 시각 기준 1회만 발송 (기존과 동일)
- 기간 일정 + 반복(repeatType)은 동시 설정 불가 — 기간 일정 선택 시 반복은 `none`으로 강제
- 기존 단일 날짜 일정의 동작이 변경되지 않음 (scheduledEndAt = null)
- 자정 플래그 리셋에서 morningNotified 리셋 로직이 `lastMorningNotifiedDate` 방식으로 변경됨

---

## Out of Scope

- 캘린더 뷰 구현 (Phase 3에서 진행)
- 컬러 라벨, 우선순위 등 UX 개선 (Phase 2)
- 기간 + 반복 조합 지원 (구현 복잡도 대비 실용성 낮음)
- 기존 데이터 마이그레이션 (scheduledEndAt이 nullable이므로 불필요)

## Tasks

**Task 1. DB 스키마 변경**

`shared/schema.ts`의 personalTasks 테이블에 다음 변경을 적용한다:

- `scheduledEndAt` 컬럼 추가 (text, nullable, default null) — 종료일시
- `morningNotified` 컬럼을 boolean에서 제거하고, `lastMorningNotifiedDate` 컬럼 추가 (text, nullable, default null) — 마지막으로 모닝 알림을 발송한 날짜(YYYY-MM-DD)
- `insertPersonalTaskSchema`에 scheduledEndAt을 optional로 추가
- Drizzle migration 파일 생성

**Task 2. API 변경 (server/routes.ts)**

POST /api/personal-tasks (생성):
- 요청 본문에서 `scheduledEndAt`을 optional로 수신
- validation: scheduledEndAt이 있을 때 scheduledEndAt >= scheduledAt인지 검증, 위반 시 400 에러
- scheduledEndAt이 있을 때 repeatType을 `none`으로 강제 (클라이언트가 다른 값을 보내도 무시)
- taskData에 scheduledEndAt 포함

PATCH /api/personal-tasks/:id (수정):
- updates에 scheduledEndAt 필드를 optional로 수신
- 기간 일정으로 변경 시 repeatType을 `none`으로 강제
- scheduledEndAt을 명시적으로 null로 보내면 단일 날짜 일정으로 복귀 가능

GET /api/personal-tasks (조회):
- 반환 데이터에 scheduledEndAt이 자동으로 포함됨 (스키마에 추가되면 별도 작업 불필요)

**Task 3. 스케줄러 변경 (server/scheduler.ts)**

sendDailyDigest() 내부의 변경:

② 개인일정 모닝 푸시 — 기간 일정 처리:
- 기존 조건 `taskDateKST > today` 대신, 기간 일정은 `today`가 `scheduledAt ~ scheduledEndAt` 범위 안에 있으면 대상에 포함
- `morningNotified` boolean 대신 `lastMorningNotifiedDate`와 오늘 날짜를 비교하여 당일 미발송 건만 처리
- 푸시 발송 후 `lastMorningNotifiedDate = today`로 업데이트 (boolean true 대신)

collectDailyDigestForUser() — 오늘/내일 일정 수집:
- 오늘 일정: `scheduledAt 날짜 <= today <= (scheduledEndAt 날짜 ?? scheduledAt 날짜)` 조건으로 변경
- 내일 예정: `scheduledAt 날짜 <= tomorrow <= (scheduledEndAt 날짜 ?? scheduledAt 날짜)` 조건으로 변경
- 단일 날짜 일정(scheduledEndAt = null)은 기존과 동일하게 동작 (`scheduledAt 날짜 === today`)

checkPersonalTasksReminder() — 10분 전 리마인더:
- 기존 로직 유지 (scheduledAt 시각 기준 1회만 발송)
- `reminderNotified`는 boolean 유지 (기간과 무관하게 시작 시각 기준 1회)

공유 일정 생성 시 푸시 메시지 (routes.ts POST /api/personal-tasks):
- 기간 일정인 경우 날짜 표시를 `4/7 ~ 4/11` 형태로 변경
- 단일 일정은 기존 형태 유지

**Task 4. 자정 플래그 리셋 변경 (server/storage.ts)**

`resetDailyNotificationFlags()` 수정:
- 기존: `morningNotified: false` 리셋
- 변경: `lastMorningNotifiedDate` 리셋 제거 — 날짜 비교 방식이므로 자정 리셋이 불필요. 매일 `lastMorningNotifiedDate !== today`인 건만 발송하므로 자동으로 다음날에 다시 발송됨
- `reminderNotified: false` 리셋은 기존대로 유지

**Task 5. 프론트엔드 변경 (client/src)**

일정 생성/수정 폼:
- "기간 설정" 토글 추가 — 활성화하면 종료일 입력 필드가 나타남
- 종료일은 시작일 이후만 선택 가능 (date picker 제한)
- 기간 설정 활성화 시 반복(repeatType) 선택을 비활성화하고 `none`으로 고정
- 기간 설정 비활성화 시 scheduledEndAt을 null로 전송

일정 목록 필터링:
- "오늘" 탭: `scheduledAt <= today <= (scheduledEndAt ?? scheduledAt)` 조건으로 변경
- "예정" 탭: 기간 일정이 아직 종료되지 않은 경우 포함
- "완료" 탭: 기존과 동일 (completed 기준)

일정 목록 표시:
- 기간 일정에는 "4/7 ~ 4/11" 형태로 기간 표시
- 단일 일정은 기존 날짜 표시 유지

**Task 6. 동작 검증**

- 단일 날짜 일정 생성/수정/삭제가 기존과 동일하게 동작하는지 확인 (하위 호환)
- 기간 일정(4/7~4/11) 생성 후 4/7, 4/9, 4/11에 각각 "오늘 일정"에 표시되는지 확인
- 기간 일정의 6AM 다이제스트 이메일에 기간 중 매일 포함되는지 확인
- 기간 일정의 모닝 푸시가 기간 중 매일 발송되는지 확인 (lastMorningNotifiedDate 비교)
- 10분 전 리마인더가 시작일 시각 기준 1회만 발송되는지 확인
- 기간 일정 + 반복 선택 시 반복이 none으로 강제되는지 확인
- 기간 일정을 수정하여 단일 일정으로 복귀(scheduledEndAt = null) 시 정상 동작 확인
- 공유 기간 일정의 수신자에게 "4/7 ~ 4/11" 형태 푸시가 발송되는지 확인
- 서버 재시작 후 catch-up에서 기간 일정이 정상 처리되는지 확인
