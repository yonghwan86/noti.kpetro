Phase 4 — 팀 협업 기능 (최종 확정본)

실행 순서: #14 → #15 → #16
전제: Phase 3 완료 후 진행
범위: C1 팀 대시보드 + C2 댓글
제외: C3 첨부파일
원칙: 기존 personal_tasks를 source of truth로 유지하고, 협업 레이어만 추가

0. 최종 설계 원칙
0-1. 팀 대시보드는 “현재 사용자 기준 personalized team view”

팀 대시보드는 관리자용 전체 감시 화면이 아니다.
현재 로그인 사용자가 실제로 접근 가능한 협업 일정 중, 자신의 팀과 관련된 것만 보여주는 화면이다.

즉, 아래 원칙을 고정한다.

같은 팀원이 만든 일정이라도 현재 사용자에게 공유되지 않았다면 보이지 않음
내가 만든 일정은 내가 owner이므로 항상 볼 수 있지만, 팀 대시보드에는 공유 설정된 일정만 포함
외부 사용자가 내 팀에 공유한 일정은 보임
private 일정은 팀 대시보드에서 제외

이 원칙으로 가야 기존 공유 구조(shareScope, shareTeamIds, shareUserIds, isShared)와 충돌이 없다.

0-2. 새 팀 일정 테이블은 만들지 않음

team_tasks 같은 별도 테이블은 만들지 않는다.
C1은 기존 personal_tasks의 협업 subset을 요약해서 보여주는 것이다.

0-3. 댓글은 task 하위 리소스

댓글은 일정(task)에 종속된다.
첨부파일은 이번 단계에서 제외한다.

0-4. 서버 스타일은 현재 코드베이스에 맞춘다

Phase 4 서버 구현은 다음 패턴을 따른다.

requireAuth(['admin','manager','staff'])
(req as any).currentUser
storage.ts 메서드 확장
routes는 storage를 호출하는 얇은 계층 유지

req.session.userId 직접 의존, raw db 쿼리 중심 예시는 참고만 하고 최종 구현 패턴은 위 구조로 고정한다.

#14 - C1 팀 대시보드
What & Why

개인 일정 시스템은 이미 존재하지만, 팀 협업 관점에서

팀 대시보드는 현재 사용자 기준으로 접근 가능한 협업 일정만 모아 팀 관점으로 보여주는 페이지다.

14-1. 협업 일정의 최종 정의

어떤 task가 팀 대시보드에 포함되려면 아래 두 조건을 모두 만족해야 한다.

A. 현재 사용자 접근 가능

현재 사용자가 아래 중 하나여야 한다.

task owner
shareUserIds에 현재 사용자 id 포함
shareTeamIds에 현재 사용자 teamId 포함
B. 팀 관련 협업 일정

아래 중 하나여야 한다.

owner가 현재 사용자 팀 소속이고, shareScope !== 'private'
shareTeamIds에 현재 사용자 teamId 포함
제외
private 일정
같은 팀원이 만들었더라도 현재 사용자에게 보이지 않는 일정
다른 팀/다른 사용자에게만 공유된 일정

이 정의를 쓰면 “내가 만든 외부 공유 일정” edge case도 자연스럽게 정리된다.
owner本人은 자기 대시보드에서 볼 수 있지만, 같은 팀 다른 사용자는 볼 수 없다. 이게 v1의 정답이다.

export const taskComments = pgTable("task_comments", {
  id: varchar("id").primaryKey().default(sql`gen_random_uuid()`),

  userId: varchar("user_id").notNull().references(() => users.id),

export const insertTaskCommentSchema = createInsertSchema(taskComments).omit({

createdAt 주의사항 (필수 반영)

다른 툴 지적이 맞습니다.
createdAt은 DB default가 없고 insert schema에서도 omit이므로, storage.createTaskComment() 내부에서 반드시 new Date().toISOString()으로 직접 채워야 합니다. 이걸 명세에 명시합니다.

SQL
CREATE TABLE IF NOT EXISTS task_comments (
  id TEXT PRIMARY KEY DEFAULT gen_random_uuid()::text,
  task_id TEXT NOT NULL REFERENCES personal_tasks(id) ON DELETE CASCADE,
  user_id TEXT NOT NULL REFERENCES users(id),
  content TEXT NOT NULL,
  created_at TEXT NOT NULL,
  updated_at TEXT
);

CREATE INDEX IF NOT EXISTS idx_task_comments_task_id ON task_comments(task_id);
CREATE INDEX IF NOT EXISTS idx_task_comments_user_id ON task_comments(user_id);

private task
